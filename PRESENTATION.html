01 — Architecture

4 couches OS + Guard X-108

Le moteur est organisé en quatre versions successives (OS0 → OS3), chacune ajoutant une couche de capacité. Le Guard X-108 est le composant de gouvernance central qui applique les règles de blocage.

OS — 0
Langage IR
Définit le langage intermédiaire avec 7 types de nœuds et le validateur de contrat (règles R1–R10).
VALUE, STATE, READ, WRITE, FLOW, COND, LOOP
Sandbox avec isolation et rollback
Validation structurelle garantie
OS — 1
Pipeline complet
Parse → Validate → X108 check → Décision. La Gate X-108 applique le délai temporel de 108 secondes.
Intent irréversible → délai 108s obligatoire
elapsed=50s, irr=True → HOLD (+58s)
Pipeline complet exécutable
OS — 2
Métriques + Seuil
Métriques quantitatives et décision par seuil. Modèle formalisé en Lean 4 : decision(m, θ) = ACT si θ ≤ S, HOLD sinon.
Invariants D1/E2/G1/G2/G3 formalisés
Noyau ne produit jamais BLOCK (L11.3)
Fuzz 200k cas — 0 violation E2
OS — 3
Composants séparés
Séparation des composants, métriques étendues et génération de visualisations SVG des décisions.
Architecture modulaire
Métriques T_mean, H_score, A_score
Visualisation SVG des décisions
Guard X-108 — Règles de gouvernance (ordre de priorité)
# Règle 1 — Délai temporel Intent irréversible + elapsed < 108s → HOLD (délai obligatoire) # Règle 2 — Contradictions entre agents Nb contradictions ≥ seuil → BLOCK (CONTRADICTION_THRESHOLD_REACHED) # Règle 3 — Flags de risque critiques FRAUD_PATTERN | LIMIT_PRESSURE | CASH_PRESSURE → BLOCK # Règle 4 — Seuil de confiance Confiance agrégée < theta_S → HOLD # Règle 5 — Non-anticipation (déclaratif) non_anticipation = True → vérifié à la gouvernance

06 — Limites déclarées

Ce qui n'est pas encore fermé

Ce projet est honnête sur ce qu'il ne démontre pas encore. Aucune limite n'est masquée. Chaque point est documenté dans LIMITS.md.

Criticité haute
Données réelles non testées
Les pipelines Trading, Bank et Ecom tournent sur des données synthétiques. Le comportement sur des flux de production réels n'est pas couvert.
Criticité haute
Attaques adversariales avancées
Les 6 batteries Phase 15.2 ne couvrent pas le fuzzing guidé par couverture, les attaques temporelles, l'injection dans la chaîne d'agents ni l'empoisonnement de métriques.
Criticité moyenne
Consensus HTTP Docker
La démo in-process (4 threads Python) passe. La démo réseau réelle (4 nœuds FastAPI) nécessite Docker. docker-compose.yml fourni dans le repo.
Criticité moyenne
Lean 4 — audit externe
lake build passe avec 0 sorry. Mais aucun expert Lean externe n'a vérifié que les énoncés formalisent correctement les propriétés informelles.
Criticité moyenne
TLA+ — bornes restreintes
TLC vérifié avec FMax=1, TauMax=3–5. Ne couvre pas FMax≥2 ni les plages temporelles arbitrairement grandes.
Criticité faible
Performance non mesurée
Latence par décision, débit maximal et comportement sous charge ne sont pas benchmarkés. Les 51 agents tournent séquentiellement.
Criticité faible
Couverture de code
Aucun rapport coverage.py généré. Les 12 tests pytest couvrent les cas principaux mais la couverture exacte des branches n'est pas mesurée.
Criticité faible
Non-anticipation déclarative
Le flag non_anticipation = True est déclaratif. Il n'est pas vérifié formellement à l'exécution comme contrainte structurelle.
07 — Perspectives d'usage

Applications possibles

Les exemples de ce repo sont des bancs d'essai techniques, pas des produits métier. Les domaines ci-dessous sont des pistes d'application — pas des implémentations existantes.

🏦
Finance et régulation
Couche de gouvernance pour la validation de transactions, le contrôle de conformité ou l'audit automatique des décisions. Décisions traçables et vérifiables plutôt que processus opaques.
Banques · Banques centrales · Régulateurs
📈
Trading algorithmique
Garde de sécurité logique autour de stratégies automatisées : empêcher certaines décisions lors de contradictions, limiter selon des contraintes formelles, produire un historique auditable.
Hedge funds · Plateformes algo · DeFi · Crypto
💳
Paiement et e-commerce
Sentinelle logique dans les flux transactionnels : détecter et bloquer certaines transactions suspectes avant exécution, tout en laissant passer les transactions normales.
Réseaux de paiement · Marketplaces · Processeurs
🔍
Audit et cybersécurité
Journal technique vérifiable : prouver qu'une règle a été appliquée, qu'une décision a été refusée, que le code n'a pas été modifié. Facilite les audits et investigations.
Cabinets d'audit · Enquêteurs financiers · Assurances cyber
⚡
Systèmes critiques
Couche de sécurité supplémentaire pour éviter des décisions dangereuses dans des systèmes automatisés. Refus de certaines actions selon des règles strictes et vérifiables.
Infrastructures · Réseaux énergétiques · Systèmes autonomes
🎓
Recherche et enseignement
Exemple concret de moteur décisionnel avec preuves formelles multi-couches (Lean 4 + TLA+ + tests adversariaux). Reproductible et auditable par des tiers.
Universités · Laboratoires · Conférences
